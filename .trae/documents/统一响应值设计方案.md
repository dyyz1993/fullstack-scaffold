# 统一 API 响应格式方案

## 核心设计原则

### 1. 错误：统一走 AppError（全局兜底）

```typescript
// 所有错误都 throw AppError，由全局处理器统一返回格式
throw new NotFoundError('User', id)
throw new BusinessError('Cannot resolve in current state')
throw new ValidationError('Invalid input', details)

// 全局错误处理器确保格式一致
```

### 2. 成功：业务返回什么就返回什么，响应包装确保格式一致

**关键点**：业务层决定返回什么数据，路由层用响应辅助函数**包装**确保格式一致。

```typescript
// 业务层 - 返回业务数据（可以是任何结构）
async function getUser(id: string): Promise<User> {
  const user = await db.findUser(id)
  if (!user) throw NotFoundError.user(id)
  return user  // 直接返回 User 对象
}

// 路由层 - 用 success() 包装，确保响应格式一致
c.json(success(result))
// 输出: { success: true, data: User, timestamp: "..." }
```

### 3. OpenAPI 响应定义与实际响应必须匹配

```typescript
// 定义响应
const getRoute = createRoute({
  responses: {
    200: successResponse(UserSchema, 'Get user'),
    404: errorResponse('User not found'),
  },
})

// 路由处理器必须返回匹配格式
.openapi(getRoute, async c => {
  const user = await userService.getUser(id)
  return c.json(success(user))  // ✅ 返回 { success: true, data: user }
})
```

## 统一响应格式

```typescript
// 成功响应
{
  success: true,
  data: T,           // 业务返回什么就是什么
  timestamp: string
}

// 错误响应（AppError.toJSON()）
{
  success: false,
  error: string,
  code: string,
  status: number,
  details?: ...,
  timestamp: string
}
```

## 响应辅助函数

```typescript
// src/server/utils/response.ts

export function success<T>(data: T) {
  return {
    success: true as const,
    data,
    timestamp: new Date().toISOString(),
  }
}

// 用于 POST/PUT 返回 201 Created
export function created<T>(data: T) {
  return {
    success: true as const,
    data,
    timestamp: new Date().toISOString(),
  }
}

// 列表响应
export function list<T>(items: T[], count?: number) {
  return {
    success: true as const,
    data: count !== undefined ? { items, count } : { items },
    timestamp: new Date().toISOString(),
  }
}

// 无数据成功响应（用于删除操作）
export function deleted(id: string) {
  return {
    success: true as const,
    data: { id },
    timestamp: new Date().toISOString(),
  }
}
```

## 业务错误处理的最佳实践

### 业务层应该返回什么？

**原则：业务层只管业务，错误由调用方决定如何处理**

```typescript
// ✅ 正确：返回 null 让调用方决定
async function getUser(id: string): Promise<User | null> {
  const user = await db.findUser(id)
  return user ?? null  // 或者直接 return user
}

// ✅ 正确：直接返回数据
async function listUsers(): Promise<User[]> {
  return db.listUsers()
}

// ✅ 正确：返回复杂业务结果
async function createOrder(input: CreateOrderInput): Promise<{
  order: Order
  payment: PaymentInfo
  shipping: ShippingInfo
}> {
  // 业务逻辑
  return { order, payment, shipping }
}
```

### 路由层负责错误包装

```typescript
// ✅ 推荐：路由层统一处理错误包装
.openapi(getRoute, async c => {
  const user = await userService.getUser(c.req.param('id'))
  if (!user) throw NotFoundError.user(id)
  return c.json(success(user))
})

// ✅ 推荐：列表直接包装
.openapi(listRoute, async c => {
  const users = await userService.listUsers()
  return c.json(list(users))
})

// ✅ 推荐：复杂业务结果直接包装
.openapi(createRoute, async c => {
  const result = await orderService.createOrder(c.req.valid('json'))
  return c.json(created(result), 201)
})
```

### 为什么不要在业务层包装响应？

```typescript
// ❌ 不推荐：业务层包装响应（耦合）
async function getUser(id: string) {
  const user = await db.findUser(id)
  if (!user) throw NotFoundError.user(id)
  return {
    success: true,
    data: user,
    timestamp: new Date().toISOString()
  }
}

// ✅ 推荐：业务层只返回业务数据
async function getUser(id: string): Promise<User> {
  const user = await db.findUser(id)
  if (!user) throw NotFoundError.user(id)
  return user  // 直接返回 User
}
```

**原因**：

1. 业务层可能被非 HTTP 场景调用（如 RPC、CLI）
2. 响应格式是展示层 concern，不应污染业务层
3. 统一包装在路由层做，更容易维护
